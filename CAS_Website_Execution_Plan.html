Project Scope — What Exactly Will Be Built

You are rebuilding centralacademyanta.com from scratch — replacing a weak PHP site with a production-grade, full-stack school platform. The scope has three distinct tiers:

Tier 1 — Public Website
27 Public Pages
Homepage, About, Academics, 10 Facility pages, Gallery, Notice Board, Toppers, TC Verification, Admission, Contact, Downloads, Events, Calendar
Tier 2 — Admin CMS
12 Admin Modules
Notice manager, Gallery manager, Staff manager, Toppers, Downloads, Sliders, Events, TC Records, Admission enquiries, Page editor, User roles
Tier 3 — Portals
4 Role Portals
Parent portal (attendance, fees, results, documents), Staff portal (attendance marking, circulars), Office portal (TC, admissions), Student document locker
What the current site lacks that you will add: proper SEO, SSR/SSG rendering, CMS without developer access, WhatsApp notifications, online fee payment (Razorpay), PWA (add to home screen), Hindi/English bilingual toggle, AI FAQ chatbot, real image optimization pipeline, secure role-based auth.

Hard Constraints (non-negotiable)

  • Must be bilingual — Hindi + English throughout (notices, UI labels, content)
  • Must be mobile-first — majority of parents in Anta are on Android + mobile data
  • Must have sub-2.5s LCP — slow sites = bad Google ranking = fewer admissions
  • Must NOT require a developer for daily operations — office staff must manage content
  • Must be deployable for under ₹2,000/month total infra cost

Tool Workflow Integration

UI Generation
Stitch AI
Generate initial screen layouts from text prompts. Get React/HTML structure fast. Export and hand off to Cursor. Use only for visual scaffolding — not final code.
Primary Dev Environment
Cursor
Where you write, edit, run, and test all code. Reads your entire codebase. Multi-file edits. Tab autocomplete. Inline AI. This is your main work surface — 80% of build time happens here.
Deep Thinking AI
Claude (browser)
Architecture, complex debugging, full feature planning, backend logic, security decisions. Paste only the relevant snippet — never the whole project.
Fallback Editor
VS Code
Git UI, terminal sessions, large file diffs, extensions (Prisma, ESLint, Tailwind). Use when Cursor is slow or you need extension that Cursor lacks.

The Exact Loop (memorize this)

Step 1 → Stitch: Describe the page/component in plain English. Get a rough React layout.
Step 2 → Cursor: Paste Stitch output. Prompt Cursor to refactor, add TypeScript, connect to data, split into components.
Step 3 → Hit a wall? Copy ONLY the problematic function/component (not the whole file) → paste into Claude browser AI.
Step 4 → Claude fixes it: Copy the solution back → paste into Cursor → apply.
Step 5 → Test: npm run dev in Cursor terminal → verify → commit to git.
Step 6 → Repeat for next feature.
Rule: Never paste more than 100 lines into browser AI. Context beyond that degrades output quality. Extract the exact problem, not the whole file. "Here's my API route, it returns 401 on the third login attempt — here's the code" beats "here's my entire codebase."

Common Mistakes to Avoid

✕ Don't do this

Building all 27 pages before the database is set up. You'll have to redo data fetching in every page.

✓ Do this instead

Schema and API routes first. Static content pages are fine early, but dynamic pages need the data layer ready.

✕ Don't do this

Dumping your entire project folder into Claude or ChatGPT and asking "what's wrong?"

✓ Do this instead

Identify the exact file and function with the bug. Paste only those 30–80 lines. Give the exact error message.

✕ Don't do this

Committing .env.local to git. Your Railway DB URL and API keys get exposed instantly.

✓ Do this instead

Add .env.local to .gitignore immediately (create-next-app does this, but verify). Use .env.example with placeholder values.

✕ Don't do this

Using client-side fetching (useEffect + fetch) for public pages. Google can't index content that renders after JS loads.

✓ Do this instead

Use async Server Components for all public pages. Data is fetched at build time (SSG) or request time (SSR) — HTML arrives pre-rendered.

✕ Don't do this

Creating a new Prisma client instance in every API route. This exhausts database connections at scale.

✓ Do this instead

Create a singleton in lib/prisma.ts using the global pattern. Every API route imports this one instance.

✕ Don't do this

Building portals (Phase 5) before the public site (Phase 3). Portals depend on auth + DB being solid.

✓ Do this instead

Public site first. It's the highest-value deliverable — parents and Google see it. Portals are internal tools, add them after.

✕ Don't do this

Skipping TypeScript because it slows you down initially. Type errors will hunt you in production.

✓ Do this instead

Use TypeScript from day 1. Ask Cursor to "add TypeScript interfaces" to any component pasted from Stitch. Catches bugs before runtime.

✕ Don't do this

Using <img> tags directly in Next.js. You lose automatic WebP conversion, lazy loading, and responsive sizing.

✓ Do this instead

Always use Next.js <Image> component from "next/image". Add Cloudinary domain to next.config.ts remotePatterns.

Optimization & Scaling

SEO — what matters for "best school in Anta Baran" ranking

  1. Every page must export a generateMetadata() function with unique title and description. Format: "Library - Central Academy Senior Secondary School, Anta, Baran | RBSE Affiliated"
  2. Add JSON-LD structured data on homepage: SchoolType schema with address, phone, geo coordinates. Google uses this for Knowledge Panel and Maps integration.
  3. Create app/sitemap.ts — returns all public URLs dynamically (including each gallery album and notice). Auto-submits as /sitemap.xml.
  4. Register on Google Search Console within 24 hours of launch. Submit sitemap. Monitor for crawl errors.
  5. Add school to Google Maps Business Profile (free) — mention website URL centralacademyanta.com. This is the #1 driver of local search visibility.

Performance — hitting 90+ Lighthouse score

  1. Fonts: Add display: 'swap' and preload: true to next/font config. Use Noto Sans for Hindi support — it covers both Devanagari and Latin scripts.
  2. Images: All gallery images through Cloudinary with auto WebP + quality 80 + progressive. Use Next.js <Image> everywhere with sizes prop for responsive sizing.
  3. ISR (Incremental Static Regeneration): Notices and gallery pages use revalidate: 60 — cached HTML served from CDN, refreshed every 60 seconds in background. Zero DB load for public traffic.
  4. API caching: Add Cache-Control: public, max-age=60, stale-while-revalidate=300 headers to public API routes (notices, gallery, staff).
  5. Bundle size: Use dynamic imports for heavy components: const Lightbox = dynamic(() => import('../components/gallery/Lightbox')). Lightbox code only loads when user clicks a photo.
